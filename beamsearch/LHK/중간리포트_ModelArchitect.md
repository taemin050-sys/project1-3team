# 헬스잇(HealthEat) 경구약제 객체검출 — 중간 리포트

**Model Architect: 이형기 (LHK)** · 지표 `mAP@[0.75:0.95]` · 중간 발표용

---

## 0. 한 줄 요약

> **"점수를 올리기 전에, 점수를 믿을 수 있게."**
> 무작정 점수를 좇는 대신 **정직한 평가 체계**를 먼저 세우고, **한 번에 한 변수씩** 검증했다. 그 과정에서 여러 통념(더 많은 데이터·더 큰 해상도·더 큰 모델이 항상 낫다)을 데이터로 반증했고, **우리 자신의 데이터 누수까지 스스로 잡아냈다.** 현재까지 확인된 유일한 확정 레버는 **모델 용량(YOLO11n→11s, +0.025)** 이며, 실험 인프라는 **2대(맥북·맥스튜디오) 분산 병렬**로 재사용 가능하게 구축했다.

## 2. 접근 철학 (왜 이렇게 했나)

1. **정직한 평가 우선.** 검증셋은 조합 단위 GroupKFold로 분리(같은 조합이 train/val에 동시 등장 금지) → 누수 없는 mAP.
2. **한 번에 한 변수.** 데이터·모델·해상도·에폭을 동시에 바꾸면 원인 귀속 불가 → 항상 하나만 바꿔 비교.
3. **멀티머신 분산 병렬.** 맥북·맥스튜디오가 각자 실험을 돌리고, 결과를 git 버스로 병합(뒤 §3.7).

## 3. 실험 여정과 시행착오

### 3.1 베이스라인 — 다중 모델 공정 비교
동일 fold·동일 하니스로 여러 후보를 비교(무지성 모델 선택 지양). 초기 단일폴드 스냅샷:

| 트랙 | 모델 | mAP | 메모 |
| --- | --- | --- | --- |
| MPS(로컬) | **YOLO11n** | 0.686 | 최소 파라미터로 최고, turnkey |
| | RetinaNet / YOLO26n / FCOS | 0.58 / 0.48 / 0.19 | 소데이터·고정예산선 수렴 열세 |
| CUDA(Colab) | **RT-DETR-l** | 0.737 | 트랜스포머-DETR, 전체 1위 |
| | FasterRCNN(warmup) | 0.655 | MPS 0.0=발산 진단 적중, warmup으로 해결 |

> 시행착오: FasterRCNN이 MPS에서 mAP 0.0 → "버그가 아니라 학습 발산"으로 진단(검출수 0 근거), warmup·lr 조정으로 0.655 복구.

### 3.2 증강 — 합성은 크게 기여, 그러나 "물량"은 플래토
AI Hub 단일 알약을 도메인(블루그레이)에 맞춰 조합 합성. **합성 696장 추가가 큰 도약**(클래스 커버리지). 그러나 그 이상 물량·분포를 바꿔도 오르지 않음:

| 데이터 | 3폴드 평균 mAP |
| --- | --- |
| real만 | 0.726 |
| **+ 합성 696** | **0.905 (+0.18)** |
| + 합성 696+1500(자연분포) | 0.896 |
| + 합성 696+2500(균형분포) | 0.909 |

> **핵심 발견:** 합성696은 확실한 도약이지만, 그 이상 물량 증가는 전부 동률(플래토). "균형분포 강제"도 개선 아님(오히려 소폭 하락). → 데이터는 이미 충분, 다른 축이 필요.

### 3.3 평가 신뢰도 3대 함정 (이 프로젝트의 핵심 스토리)

**함정 ① 단일폴드는 순위를 못 가린다 (노이즈).**
fold0 한 폴드에선 `696=0.933 → +2500=0.917 → +1500=0.871`처럼 "데이터를 늘렸는데 하락"하는 비직관적 결과. 부트스트랩 결과 **95% 신뢰구간이 전부 겹침**(val 51장이 너무 작음). → **3폴드 GroupKFold**로 폴드평균 측정. 위 §3.2 표가 그 결과이며, "하락"은 노이즈였음이 확정(다른 폴드에선 순위 역전).

**함정 ② 대회 GT가 실알약 일부를 누락.**
합성 배경을 만들다 "알약 없는 영역"을 골랐는데 실알약이 들어감 → GT 누락 발견. 검출기+육안 교차검증으로 **13건 확정**(232장 중 13장). 무라벨 실알약을 모델이 검출하면 오탐(FP)으로 계산 → **우리 val이 약 0.004 과소평가**. 대회 원본은 무수정, 로컬 corrected 라벨셋을 별도 운용.

**함정 ③ "0.960 vs 0.933" 채점 불일치 → 지표 범위 차이.**
같은 모델·같은 val인데 두 수치. 규명 결과 하나는 **mAP@[0.50:0.95]**(표준 COCO), 하나는 **mAP@[0.75:0.95]**(대회 지표). 하니스 버그가 아니라 IoU 범위 차이. → 지표 라벨을 항상 명시, 대회 정답은 0.75:0.95로 통일.

> **교훈:** 세 함정 모두 "그럴듯한 숫자"를 의심해 파고든 결과. 평가를 믿을 수 있게 만든 뒤에야 이후 모든 실험이 의미를 가졌다.

### 3.4 모델 용량 — 현재 유일한 확정 레버 (단, 비단조)
데이터축이 막히자 **모델 체급**을 검증(전부 base 데이터·누수無):

| 모델 | 파라미터 | mAP | 메모 |
| --- | --- | --- | --- |
| YOLO11n | 2.6M | 0.905 | |
| **YOLO11s** | 9.4M | **0.930 (+0.025)** | **스윗스팟** |
| YOLO11m | 20M | 0.907 (100ep) | base에선 하락 |

> **발견:** 용량은 단조 증가가 아니다. **11s가 스윗스팟**. 11m은 base 877장에선 **데이터 기아→과적합**(에폭을 100으로 늘려도 회복 안 됨). 즉 "더 큰 모델엔 더 많은 데이터가 필요"함을 실증 — 11m의 진짜 판정은 데이터를 늘린 뒤로 보류.

### 3.5 해상도 — 폐기
"976×1280 원본을 640으로 축소하니 손해 아닐까?" → 학습·추론 해상도를 960으로. **11n·11s 양쪽에서 640보다 하락.** 해상도 레버 폐기.

### 3.6 real copy-paste 누수 발견 — 자기검증의 사례
실 알약을 잘라 새 조합으로 재합성한 "real-copy" 증강이 처음엔 최고(0.945)로 보였다. 그러나 **폴드별로 뜯어보니** 이득이 한 폴드(fold1)에만 몰림. 데이터의 `source_file`을 폴드에 직접 매핑해 검증한 결과:

- real-copy 소스 = **fold0-train 181장**(= folds 1·2·3·4 조합)
- k-fold 평가 시 **fold1·fold2의 val 이미지 알약이 그대로 train에 재등장 = 누수 확정**
- **누수 없는 fold0에선 real-copy가 오히려 소폭 하락** → "+0.015 게인"은 누수 산물

> **결론:** "real-copy = 확정 레버"는 **누수 오염 결과였고 무효화**했다. 대응으로 **폴드별 누수안전 소스 매니페스트**를 만들어 데이터엔지니어(Codex)에게 재생성 요청, 재검증 예정. — 남의 결과가 아니라 **우리 결과를 스스로 의심해 잡아낸 사례.**

### 3.7 분산 작업 인프라 — 재사용 가능한 경량 시스템
맥북↔맥스튜디오 병렬 작업을 위해 **개인 툴킷(private repo `lab-kit`)** 구축:
- **git을 버스로:** 코드·결과(작음)만 공유, 데이터·가중치(무거움)는 각 머신 로컬.
- **머신별 결과 파일** → 동시 실행해도 병합 충돌 0. **경로 자동감지**로 머신별 데이터 위치 차이 흡수.
- 파라미터화된 실험 러너(모델·해상도·에폭·데이터를 환경변수로) → 스크립트 난립 없이 재사용.
- 프로젝트 무관 설계 → 다른 프로젝트에도 재사용 가능.

## 4. 핵심 결과 종합 (3폴드 평균, 누수無·대회지표)

```
base(real만)        0.726
 + 합성696          0.905   ← 합성의 도약(+0.18)
 + 합성 물량↑        0.90대  ← 플래토(물량은 레버 아님)
11n → 11s           0.930   ← 용량(+0.025), 현재 최고·유일 확정 레버
11s → 11m(base)     0.907   ← 데이터 기아(더 많은 데이터 필요)
해상도 960          하락     ← 폐기
real-copy           보류     ← 누수 발견, 재검증 대기
```

**현재 최고 신뢰 config: YOLO11s + (real232 + 합성696) @640 = 0.930.**

## 5. 시행착오가 준 교훈

1. **작은 val(51장)의 단일 수치는 믿지 않는다** — 멀티폴드 평균으로만 판단.
2. **더 많은/더 큰 것이 항상 낫지 않다** — 합성 물량·해상도·모델체급 모두 반례를 만남.
3. **데이터의 출처(source)를 끝까지 추적한다** — GT 누락·real-copy 누수 모두 "어디서 왔나"를 파서 발견.
4. **우리 결과를 남의 결과보다 더 의심한다** — 가장 좋아 보인 real-copy가 사실 누수였다.
5. **인프라가 속도다** — 분산·자동화로 여러 축을 병렬 검증.

## 6. 현재 상태 & 다음 단계

**상태:** 평가 체계 확립 완료. 확정 레버 = 용량(11s). real-copy는 누수 재검증 대기, 11m은 데이터 확장 후 재판정 대기.

**다음:**
1. **누수 없는 per-fold real-copy** 재생성(진행 중) → real-copy 정직 재판정.
2. **용량 + 충분한 데이터** 결합(11m을 늘린 데이터로 재검증).
3. **아키텍처 다양성(RT-DETR)** 확보 → 앙상블·XAI.
4. 최종 레시피 확정 → **real 232 전량 + 최적 증강**으로 학습해 제출.

## 7. 맺음

이 프로젝트의 성과는 특정 점수가 아니라 **"믿을 수 있는 결론을 내는 과정"** 자체다. 통념을 데이터로 검증하고, 노이즈·누락·누수를 걸러내며, 우리 자신의 실수까지 잡아냈다. 남은 기간에도 같은 규율로 **용량·데이터·다양성** 세 축을 정직하게 채워 최종 모델을 확정한다.
