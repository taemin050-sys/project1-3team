## 0. 문서 개요

문서 A가 *데이터*, B가 *서빙·배포*를 정의했다면, C는 **사용자가 보는 것**을 정의한다. 세 축으로 서비스면을 완결한다.
1. **화면·플로우·접근성** — 누가(고령자 포함) 어떻게 쓰는가 (§1~7)
2. **의료 디스클레이머·프라이버시** — 헬스케어 인접의 책임 있는 설계 (§6)
3. **발표·데모 스크립트** — 서사 스파인을 라이브로 증명 (§8~9)

> 범위 제외: 데이터 사전(A), API·인프라(B).

## 1. UX 디자인 원칙

문제정의가 지목한 **핵심 사용자는 고령자·만성질환자**다. 따라서 미감보다 **신뢰·명료·접근성**이 1순위.

| # | 원칙 | 적용 |
| --- | --- | --- |
| P1 | **신뢰 우선** | 결과에 신뢰도·디스클레이머를 항상 동반, 과장 없는 톤 |
| P2 | **명료함** | 한 화면 한 목적, 전문용어 최소화, 큰 타이포 |
| P3 | **고령자 접근성** | 큰 터치 타깃·고대비·쉬운 말 (§7, WCAG 2.2 AA) |
| P4 | **안심 톤** | 경고하듯이 아니라 차분하게 안내 |
| P5 | **실패에 관대** | 빈 결과·오류도 다음 행동을 친절히 제시 |

> 비주얼: 깨끗하고 고대비인 라이트 테마 권장(고령자 가독성). 포트폴리오 일관성을 원하면 기존 디자인 토큰(Pretendard Variable + JetBrains Mono, WCAG 2.2 AA) 적용 가능.

## 3. 화면 설계 (와이어프레임)

### 3.1 랜딩 / 업로드
```
┌──────────────────────────────────────────┐
│  💊 헬스잇 알약 인식                          │
│                                            │
│  복용 중인 약을 촬영하면                       │
│  무슨 약인지 알려드려요                        │
│                                            │
│  ┌────────────────────────────────┐        │
│  │   📷  사진 올리기 / 여기로 끌어다 놓기 │        │
│  │        (또는 촬영하기)            │        │
│  └────────────────────────────────┘        │
│                                            │
│  예시로 해보기:  [ 샘플 1 ] [ 샘플 2 ] [ 샘플 3 ] │  ← 데모 보험(B §8)
│                                            │
│  ⚠️ 참고용 서비스입니다. 복약 전 약사·의사와        │  ← 상시 디스클레이머
│     상담하세요.                              │
└──────────────────────────────────────────┘
```

### 3.2 처리중
```
┌──────────────────────────────────────────┐
│            약을 인식하고 있어요…             │
│              ◴  (로딩)                      │
│           잠시만 기다려 주세요               │
└──────────────────────────────────────────┘
```

### 3.3 결과 (핵심 화면) ★
```
┌──────────────────────────────────────────┐
│  인식 결과 — 약 3개를 찾았어요               │
│  ┌────────────────────┐                    │
│  │  원본 + 박스 오버레이  │   ①②③ 번호 배지     │
│  │  ① ② ③              │   (색+숫자 병기)     │
│  └────────────────────┘                    │
│                                            │
│  ①  ○○정              신뢰도 91% ▰▰▰▰▱  [자세히] │
│      성분 △△ · 일반의약품 · 원형 · 하양 · 각인 ABC │
│  ──────────────────────────────────────────  │
│  ②  □□캡슐            신뢰도 78% ▰▰▰▱▱  [자세히] │
│      성분 ◇◇ · 전문의약품 · 캡슐 · 노랑          │
│  ──────────────────────────────────────────  │
│  ③  미상 약품          신뢰도 낮음        [자세히] │  ← fallback(A §6.2)
│                                            │
│  ⚠️ AI 추정 결과입니다. 정확한 복약 정보는          │
│     반드시 약사·의사에게 확인하세요.             │
│  [ 다시 시도 ]      [ 결과가 틀렸나요? ]          │  ← 피드백(심화)
└──────────────────────────────────────────┘
```

### 3.4 약 상세 (확장 카드)
- `drug_master` 전체: 제품명·성분·분류·전문/일반·모양·색상·각인(앞/뒤)·크기·제조사·허가일자·대표이미지
- 상단에 디스클레이머 재노출

### 3.5 빈 결과
```
약을 찾지 못했어요.
더 밝은 곳에서, 약이 화면에 잘 보이도록 다시 촬영해 주세요.
[ 다시 시도 ]
```

### 3.6 오류
```
처리 중 문제가 발생했어요. 잠시 후 다시 시도해 주세요.
[ 다시 시도 ]
```

## 4. 핵심 컴포넌트 명세 (+ 데이터 매핑)

| 컴포넌트 | 설명 | 데이터 출처(A) |
| --- | --- | --- |
| 업로드 드롭존 | 클릭·드래그·촬영, 타입·크기 안내 | — |
| 샘플 버튼 | 내장 이미지로 즉시 시연(라이브 의존 제거) | 데모 자산 §9 |
| 검출 오버레이 | 원본 위 bbox + 번호 배지(색+숫자 병기) | `prediction_item.bbox` |
| **약 정보 카드** | 제품명·성분·분류·전문/일반·모양·색상·각인·제조사 | **`drug_master`(M) 컬럼** |
| 신뢰도 표시 | %·게이지, *색만으로 구분 금지* | `prediction_item.score` |
| 디스클레이머 배너 | 상시 노출(랜딩·결과·상세) | §6 |
| 피드백 위젯(심화) | "결과가 틀렸나요?" → 교정 | `feedback`(A) |

> 카드 필수 필드 = 문서 A §5.1의 `(M)` 9종. "category 24"가 아니라 **실제 약 정보**가 뜨는 지점.

## 5. UX 라이팅 / 마이크로카피

톤: **차분·명료·안심**. 전문용어는 카드 안에만, 안내문은 쉬운 말.

| 맥락 | 카피 |
| --- | --- |
| 업로드 유도 | "복용 중인 약을 촬영하면 무슨 약인지 알려드려요" |
| 처리중 | "약을 인식하고 있어요…" |
| 결과 헤더 | "약 N개를 찾았어요" |
| 빈 결과 | "약을 찾지 못했어요. 더 밝은 곳에서 다시 촬영해 주세요." |
| 오류 | "처리 중 문제가 발생했어요. 잠시 후 다시 시도해 주세요." |
| 신뢰도 낮음 | "정확도가 낮아요. 약사·의사 확인을 권해요." |
| 피드백 | "결과가 틀렸나요?" |

## 6. 의료 디스클레이머 · 프라이버시 (user-facing) ★

헬스케어 인접 서비스의 책임 설계. **상시 노출** + 결과·상세에 재노출.

### 6.1 짧은 배너 (상시)
> ⚠️ 본 서비스는 알약 식별을 돕는 **참고 도구**이며, 의료 조언이나 전문가 상담을 대체하지 않습니다. AI 추정 결과는 부정확할 수 있으니 **복약 전 반드시 약사·의사에게 확인**하세요.

### 6.2 전체 고지 (about·푸터)
> **의료 면책 고지**
> 본 서비스가 제공하는 약 정보는 AI 모델의 추정 결과로, 실제와 다를 수 있습니다. 진단·처방·복약 결정의 근거로 사용하지 마시고, 정확한 정보는 의약 전문가에게 확인하시기 바랍니다. 응급 상황에서는 즉시 의료기관에 연락하세요.
>
> **개인정보 안내**
> 업로드하신 이미지는 알약 인식 처리에만 사용되며 **별도로 저장되지 않습니다**. (서비스 개선용 익명·집계 로그만 수집)

### 6.3 노출 규칙
| 위치 | 노출 |
| --- | --- |
| 랜딩 | 짧은 배너 |
| 결과·상세 | 짧은 배너 재노출(결과 바로 아래) |
| 푸터/about | 전체 고지 |
| 신뢰도 낮음 | 추가 경고 문구 |

> 원본 이미지 미저장은 문서 B §9·문서 A §9와 일치(해시만 로깅).

## 7. 접근성 (WCAG 2.2 AA + 고령자) ★

문제정의의 고령자 사용자와 직결되는 핵심 품질.

| 항목 | 기준 |
| --- | --- |
| 색 대비 | 본문 ≥ 4.5:1, 큰 글씨 ≥ 3:1 |
| 타이포 | 본문 ≥ 16px, 핵심 정보(약 이름) 더 크게 |
| 터치 타깃 | ≥ 44×44px (고령자 고려 여유 있게) |
| 색 의존 금지 | 박스·신뢰도는 **숫자/라벨 병기**(색맹 대응) |
| 키보드 | 전 기능 키보드 접근·가시 포커스 |
| 대체텍스트 | 이미지 alt·ARIA 라벨 |
| 쉬운 말 | 안내문은 비전문 용어, 전문용어는 카드 안으로 |

## 8. 발표 · 데모 스크립트 ★

최종 발표 **15분 + Q&A 5분**(전원 참여). 서사 스파인 중심으로 라이브 데모를 클라이맥스에 배치.

> **스파인:** *"우리는 리더보드 점수가 아니라 환자가 쓸 수 있는 서비스를 만들었고, 그 과정에서 점수도 따라왔다."*

### 8.1 발표 아크 / 타이밍 (= 보고서 골격 겸용)

| # | 구간 | 시간 | 핵심 |
| --- | --- | --- | --- |
| 1 | 문제 (Hook) | 1.5분 | 고령자·만성질환자의 약 식별 어려움(문제정의) |
| 2 | 우리가 만든 것 | 1분 | **서비스를 만들었다**는 테제를 먼저 제시 |
| 3 | 데이터의 진실(EDA) | 2분 | 저데이터·불균형·고-IoU, 구조적 발견 |
| 4 | 방법(과정) | 3분 | **빔서치 협업 루프** + 실험 곡선, 무엇이 통했나 |
| 5 | **라이브 데모** | 3.5분 | 업로드→박스→**약 정보 카드**(클라이맥스) |
| 6 | 결과·발견 | 2분 | 최종 점수 + **정직한 천장 분석** |
| 7 | 서비스·임팩트·로드맵 | 1.5분 | 배포·비즈니스 가치·피드백 루프·병용금기(향후) |
| 8 | 클로징 | 0.5분 | 스파인 재진술 |

### 8.2 라이브 데모 스텝 (#5 상세)

```
[사전] 발표 직전 워밍업 완료(/health 그린, 샘플 추론 1~2회)  ← B §8
 ①  "실제로 보여드리겠습니다" — 라이브 URL 오픈
 ②  [샘플 N] 버튼 클릭 (라이브 업로드 의존 제거, 가장 안전)
 ③  박스 오버레이 + 약 카드 등장
 ④  강조: "category 24가 아니라 제품명·성분·전문/일반까지 나옵니다"
 ⑤  신뢰도·디스클레이머 가리키며 — "책임 있게 설계했습니다"
 ⑥  (옵션) 어려운 케이스 하나 — 정직하게 한계도 보여줌
```

### 8.3 Q&A 대비 (예상 질문 → 답변 핵심)

| 예상 질문 | 답변 핵심 |
| --- | --- |
| 왜 이 지표/모델? | 고-IoU(0.75:0.95) → localization 중심, 단일+앙상블 이중 트랙 |
| 저데이터 극복? | 전이학습·증강·외부데이터, 빔서치로 효과 검증 |
| 천장 클래스는? | 데이터 구조적 한계 솔직 공유(서사 신뢰도) |
| 오인식 위험? | 디스클레이머·신뢰도 노출·피드백 루프 |
| 실서비스 가능성? | 동작 데모 + 한계·로드맵(병용금기·재학습) |
| 데이터 편향? | 단일 데이터 계약·병합 전 분포 점검(A) |

### 8.4 역할 분담(발표)
전원 참여 — 예: 문제·서사(이태민), 데이터(홍우석), 모델·방법(이형기), 실험·결과(장한빈), **라이브 데모·서비스(최중열)**. *(팀 합의로 조정)*
