# 실행 전략 (Execution Strategy) — 헬스잇 경구약제 객체 검출

> **코드잇 스프린트 AI 엔지니어링 12기 · 초급 프로젝트 · 3팀 (ULTRA CAPSHYONG ITEM WITH 4 VALUES)**
> 문서 계보: 팀 계획서 → SRS → PRD → **〔본 문서〕 실행 전략**
> 작성: 이형기(Model Architect) · 작성일 2026-06-26 · v0.1
> 성격: *"어떻게 이길 것인가"* — 팀 운영 모델 + 6대 승리 차원의 동시 최적화

## 0. 이 문서가 답하는 질문

PRD가 *무엇을 어떻게 만들지*를 정의했다면, 본 문서는 **"한정된 2.5주에, 전원이 풀스택으로 참여하면서, 어떻게 성능·모델·산출물·과정·서사·서비스를 동시에 최상으로 끌어올릴 것인가"**에 답한다.

> **운영 대전제 (팀 합의):** 역할은 분담했으나 **모든 단계에 전원 참여**한다. 각자 *전 과정을 주도하고 구현한다*는 마음으로 임하되, 단계마다 산출물을 공유하고 **우수한 1~3개를 선별해 다음 단계로 발전**시키며, 결과에 따라 **제외**하거나 새 통찰에 따라 **신규 도입**하는 **기민한(agile)** 방식으로 진행한다.

## 1. 승리의 정의 (6대 차원)

점수 하나가 아니라 **여섯 개 전선에서 동시에 이긴다.** 각 차원은 서로를 강화하도록 설계한다.

| # | 승리 차원 | 무엇이 "이긴 것"인가 |
| --- | --- | --- |
| ① | **성능** | mAP@[0.75:0.95] 우상향 곡선 + *각 향상의 원인 설명* |
| ② | **모델** | 견고하고 재현 가능한 검출 모델 (+ 서빙 가능한 경량 버전) |
| ③ | **산출물** | 재현 코드·보고서·협업일지가 일관된 완결성으로 연결 |
| ④ | **과정 우수성** | 빔서치 협업 루프 자체가 모범 사례가 됨 |
| ⑤ | **서사** | "점수를 쫓지 않고 *서비스*를 만들었더니 점수가 따라왔다" |
| ⑥ | **웹 서빙·배포** | 촬영→식별→약 정보까지, **실제 동작하는 웹 서비스** 데모 |

> 차원 간 시너지: ⑥ 서비스를 위해 만든 **약 정보 테이블**(§6)이 ⑤ 서사의 클라이맥스가 되고, ④ 과정 기록(decision log)이 ③ 보고서의 뼈대가 되며, ② 서빙 경량 모델이 ① 추론 속도(NFR-03)를 충족한다.

## 2. 팀 운영 모델 — 빔서치 협업 루프 (Explore → Share → Select → Advance)

전원이 매 단계 병렬로 탐색하고, 동일 기준으로 평가해 **상위 1~3개(beam width)** 만 살려 다음 단계로 보낸다. 빔은 발전(Advance)·제외(Prune)·주입(Inject)으로 매 사이클 갱신된다. 누구나 풀스택이므로 **어떤 빔이든 이어받아 발전**시킬 수 있다.

```
   ┌──────────── 1 사이클 (≈데일리~페이즈 게이트) ─────────────┐
   │                                                          │
   │  EXPLORE        SHARE          SELECT         ADVANCE     │
   │  발산           공유            수렴           발전/제외/주입 │
   │  전원 병렬 시도 → 동일 평가기준 → fitness 게이트 → top 1~3 진출 │
   │  (데이터·제안·    공유 (로컬 mAP    로 선별         + 신규 아이디어 │
   │   모델·실험)      하니스 §PRD5.6)                  주입(Inject)  │
   │       ▲                                              │       │
   │       └──────────────── 다음 사이클로 ◀───────────────┘       │
   └──────────────────────────────────────────────────────────┘
```

### 2.1 선별 게이트 (Fitness Function)

빔으로 살아남으려면 **통과 조건(gate)** 을 전부 만족하고, **점수(rank)** 로 상위 1~3에 들어야 한다.

| 구분 | 기준 |
| --- | --- |
| **통과 조건(필수)** | ⓐ config로 **재현 가능** ⓑ 제출 포맷 검증 통과(IR-06) ⓒ val/public 괴리 비합리적이지 않음 |
| **랭킹 1순위** | 로컬 val **mAP@[0.75:0.95]** (PRD §5.6 하니스로 통일 측정) |
| **랭킹 보조** | 클래스별 AP 균형 · 추론 속도(서빙 고려) · **탐색 가치(신규성)** |

> 보조 기준의 "신규성"이 중요하다 — 점수가 약간 낮아도 *새로운 방향을 여는 빔*은 1개 슬롯을 배정해 다양성을 유지한다(국소 최적 탈출).

### 2.2 Keep / Kill / Inject 규칙

| 결정 | 트리거 | 처리 |
| --- | --- | --- |
| **Keep(발전)** | 게이트 통과 + 랭킹 상위 1~3 | 다음 단계로, 변주 심화 |
| **Kill(제외)** | 2사이클 연속 무개선 or 게이트 실패 | 보류(log에 사유 기록) |
| **Inject(주입)** | EDA/실험서 새 통찰·외부 기법 발견 | 빔 1슬롯 신규 배정 |

### 2.3 운영 규율
- **Beam width 1~3 엄수** — 너무 많은 빔은 모두를 얕게 만든다(역량 분산 리스크 대응).
- **동일 평가기준** — 모든 빔은 PRD §5.6 로컬 하니스로만 비교(주관 배제).
- **Decision Log 상시 기록** — `사이클 · 빔 · 결정(Keep/Kill/Inject) · 근거 · 지표`. 이 로그가 ③산출물·④과정·⑤서사의 1차 사료(史料)가 된다.
- **컴퓨트 예산 배분** — 한 빔이 자원을 독점하지 않도록 데일리 브리핑서 GPU/제출 10회를 분배.

### 2.4 마일스톤 게이트 & 레포 승격 워크플로우

작업 단위는 **팀원별 개인 이름 레포**다. 매 단계 팀 공통 레포에 통합하지 않고, **마일스톤까지 개인 레포에서 끝까지** 진행한 뒤 결과물을 비교·평가해 우수안만 **팀 레포로 승격(promote)** 한다. 승격은 **전원 평가**로 결정한다.

| 페이즈 | 개인 레포에서 진행 | 승격 게이트(전원 평가) |
| --- | --- | --- |
| **P1 탐색** | 데이터수집 → 전처리 → 분석(EDA) → 모델링 → 학습 → 평가 → 종합리포트까지 완주 | 결과 비교·순위 → 우수안 팀 레포 승격 |
| **P2 고도화** | 승격본을 개인 레포에 **디폴트로 가져와** 각자 고도화 | 성능·속도 종합평가 → **최종 2모델**(고성능 1 + 서빙 경량 1) 승격 → **1차 마감** |
| **P3 서빙·배포** | 서비스 개발은 **개별 전담**(각자 본인 계획·설계대로). 최중열은 본인 설계로 개인 레포 진행 | 두 서비스 결과물 시연 비교 → 전원 평가로 1개 선정 승격 |
| **최종** | — | 최종 결과물 **종합 시연 및 발표** |

```
개인 레포(완주) → [전원 평가·순위] → 팀 레포 승격 → 개인 레포로 디폴트 회수
   → 개인 고도화 → [성능·속도 평가] → 최종 2모델 승격(1차 마감)
   → 개별 서비스 개발 → [시연 비교 평가] → 서비스 1종 승격 → 최종 발표
```

> 원격 반영·승격은 **사람(전원) 주도**. 개인 레포에서 마일스톤을 완주한 산출물만 게이트에 올린다. (빔서치 루프 §2가 이 승격 게이트 위에서 돈다.)

## 3. 핵심 전략 원칙 (회고 자산 기반)

그동안의 미션·대회 회고에서 검증된 원칙을 이 프로젝트에 적용한다.

1. **EDA-First.** 모델 이전에 데이터 구조를 먼저 읽는다. 클래스 분포·bbox 통계·메타데이터로 *어디에 헤드룸이 있고 어디가 천장인지* 먼저 지도화한다.
2. **"학습 예산이 병목이다"** *(Mission 8 교훈).* 이국적 아키텍처에 시간을 태우기 전에 **전이학습·해상도·데이터·에폭** 같은 예산 변수를 먼저 확보한다. 아키텍처 교체는 그 다음이다.
3. **"데이터 구조적 천장을 인지한다"** *(Mission 9 교훈).* 특정 클래스가 데이터 한계로 막혀 있으면 거기서 과최적화를 멈추고 **헤드룸 있는 곳으로 자원을 옮긴다.** 천장은 솔직하게 보고서·발표에 쓴다(=서사 신뢰도).
4. **Baseline-First, Fast Loop.** 1주차에 파이프라인을 관통(E1)시켜 floor를 확보한 뒤 개선한다.
5. **앙상블은 마무리 수(手)** *(Food-101 교훈).* 단일 모델을 충분히 끌어올린 뒤 **WBF로 마지막 이득**을 취한다. 근접 성능일 때 균등가중이 강하다는 점을 가중탐색의 출발점으로.
6. **재현성은 토대.** seed 고정·config·native eval transform 정합 — 재현 안 되는 점수는 빔이 될 수 없다.

## 4. 단계별 실행 전략 (루프 × 타임라인)

각 페이즈는 한 번 이상의 빔서치 사이클을 돈다.

| 페이즈 | 기간 | EXPLORE(발산) | SELECT 게이트 산출 | 진출(beam) |
| --- | --- | --- | --- | --- |
| **P1 기반** | 6/26~7/1 | 전원 EDA + 데이터 계약 후보 + **E1 베이스라인** | 동작하는 파이프라인 + floor 점수 | 데이터 계약 1개 + 베이스라인 1~2개 |
| **P2 개선** | 7/2~7/6 | 증강·해상도·모델군(E2~E4) 병렬 | val mAP 향상 빔 | 모델·전처리 조합 1~3개 → **중간발표** |
| **P3 고도화** | 7/8~7/12 | 외부데이터·TTA·WBF·후처리(E5~E8) + **서빙 트랙 병행** | 최고 점수 빔 + 서빙 경량 빔 | 대회용 1~2개 + 서빙용 1개 |
| **P4 마감** | 7/13 | 최고 빔 freeze, 보고서·서사 정리, 데모 확정 | 제출본 + 배포 데모 | 최종 1개 (+ 데모 URL) |

> P3에서 **대회 트랙과 서빙 트랙을 분기**한다(§5②). 중간발표(7/7)는 P2 종료 게이트로 활용 — 외부 피드백을 Inject 소스로 삼는다.

## 5. 차원별 전략

### ① 성능 (Kaggle mAP@[0.75:0.95])
- 고-IoU 게임 → **localization 우선**: 해상도 상향(E3) · 2-stage 비교(E4) · WBF(E7) · score threshold 스윕(E8).
- **로컬 하니스로 제출 전 선별**(10회/일 절약). 클래스별 AP로 취약 클래스 표적 개선, 단 *데이터 천장 클래스는 과투자 금지*.

### ② 모델 (이중 트랙 전략 ★)
경쟁 최적 모델과 서비스 최적 모델은 **목적이 다르다.** 둘을 분리 운용한다.

| 트랙 | 목표 | 구성 | 지표 |
| --- | --- | --- | --- |
| **대회 트랙** | 최대 정확도 | 앙상블(YOLO+FRCNN, WBF), TTA | mAP 최대화 |
| **서빙 트랙** | 실시간성·경량 | **단일 강한 모델**(예: best YOLO) | 정확도/지연 균형(NFR-03) |

> 무거운 앙상블을 웹에 올리면 느리다. 서빙엔 단일 모델, 대회엔 앙상블 — 이 분리가 ①과 ⑥을 동시에 만족시킨다.

### ③ 산출물
- Decision Log(§2.3) → 보고서 골격으로 직결. 재현 코드(config 단위), README 재현 절차, 협업일지 일관성.
- "정리는 마지막에"가 아니라 **매 사이클 기록 → 마지막에 편집**.

### ④ 과정 우수성
- 빔서치 협업 루프 자체를 **방법론으로 명문화**해 발표에 제시. "역할 분담을 넘어 전원 풀스택으로, 빔서치로 수렴시켰다"는 운영 모델이 차별점.

### ⑤ 서사 (Narrative)
- **스파인(thesis) 한 문장:** *"우리는 리더보드 점수가 아니라 환자가 쓸 수 있는 서비스를 만들었고, 그 과정에서 점수도 따라왔다."*
- 아크: EDA가 드러낸 데이터의 진실(저데이터·불균형·고-IoU) → 베이스라인 floor → 빔서치로 체계적 개선 → **핵심 발견(무엇이 통했고 어디가 천장이었나, 정직하게)** → 앙상블 마무리 → **동작하는 웹 서비스로 비즈니스 가치 실현**.
- 차별화: 대부분 팀이 *점수 곡선*을 보여줄 때, 우리는 *배포된 서비스 + 규율 있는 과정 + 정직한 천장 분석*을 함께 보여준다.

## 6. 웹 서빙·배포 전략 (실전 서비스화)

### 6.1 서비스 인텔리전스 레이어 — 데이터 자산의 재발견 ★
제공 데이터의 COCO 메타데이터에는 **실제 약 정보**가 들어 있다: `dl_name`(제품명), `dl_name_en`, `dl_material`(성분), `di_class_no`(약품분류), `di_etc_otc_code`(전문/일반), `drug_shape`, `color_class`, `dl_company`(제조사) 등.

→ **`category_id → 약 정보 카드` 테이블**을 구축한다(DR-04 매핑의 자연스러운 확장). 모델이 "category 24"가 아니라 **"○○정 / 성분 △△ / 일반의약품 / 제조사 □□"** 를 반환 → 헬스잇의 비즈니스 가치(복약 정보 제공)를 실제로 구현. *이것이 데모의 클라이맥스이자 ⑤서사의 핵심.*

### 6.2 배포 티어 (점진적 야심)

| 티어 | 범위 | 스택 | 용도 |
| --- | --- | --- | --- |
| **T1 (MVP·필수)** | 이미지 업로드 → 박스 오버레이 + 약 정보 카드 | **Gradio/Streamlit + Hugging Face Spaces** | 발표용 **라이브 데모 URL** (가장 빠른 경로) |
| **T2 (심화)** | 정식 웹앱 UI | **FastAPI(추론 API) + Next.js(Vercel)** | 실서비스형, 최중열 서빙 트랙 |
| **T3 (확장)** | 병용금기 스텁·복약 관리 | 룰 테이블/DB | 향후 로드맵(범위 외 명시) |

> 권장 경로: **T1을 7/10까지 확보**(발표 데모 보장) → 여력 시 T2. T1은 서빙 트랙 단일 모델(§5②)을 그대로 올리면 된다.

### 6.3 서빙 파이프라인
